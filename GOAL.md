# GOAL.md — VR自室プロジェクト「myroom-vr」(仮称)

このドキュメントはClaude Code(Fable 5)へのゴール設定書。これ一枚を正として、一気通貫で実装すること。

---

## 1. ゴール

Meta Quest のブラウザから入れる、自分だけのVRの部屋を作る。

- **リアルでリッチな質感**:PBRテクスチャ+HDRI環境光による実在感。「ゲームっぽいローポリ」ではなく「写真的な落ち着き」を目指す
- **屋内と屋外がつながる**:書斎 → 縁側 → 庭 がシームレスに歩ける一軒の空間
- **仕掛けのある空間**:触れる・切り替わる・鳴る要素を随所に配置(後述のP0/P1リスト)
- **完成の定義**:Questブラウザで入室し、P0仕掛けが全て動き、体感が滑らか(§8の受け入れ基準)

## 2. 技術選定(決定事項・変更不可)

| 項目 | 決定 | 理由 |
|---|---|---|
| プラットフォーム | **WebXR**(Questブラウザで閲覧) | コードのみで完結、Unity手作業不要 |
| エンジン | **Three.js**(vanilla)+ Vite | 一気通貫実装に最適、依存最小 |
| 言語 | TypeScript | |
| ホスティング | GitHub Pages(HTTPS必須のため)+ ローカル開発は `@vitejs/plugin-basic-ssl` | WebXRはHTTPS必須 |
| アセット | **CC0のみ**(Poly Haven / ambientCG)。ダウンロードスクリプトで取得しリポジトリ管理 | ライセンス問題ゼロ |
| VR入力 | WebXR標準(コントローラーレイ+トリガー) | |

VRChatへの移植は将来のフェーズ2とし、今回はスコープ外。ただしシーン構成をモジュール分割し、移植時にレイアウトを流用できる程度の分離は保つこと。

## 3. 空間コンセプトとレイアウト

**「書斎と縁側と庭がつながる家」**。和モダンの平屋。時間帯によって表情が変わる。

```
[庭]  飛び石・池 or 鹿威し・楓の木・焚き火スペース
 │ (段差なしで往来可)
[縁側] 木の濡れ縁・座布団・軒下の風鈴
 │ (障子/ガラス戸 — 開閉ギミック)
[書斎] 文机・本棚(壁一面)・掛け軸・行灯・ラジオ・畳
 │
[土間/玄関] スポーン地点。ここから空間全体を見渡せる導線
```

- 空間全体は 20m × 20m 程度に収める(パフォーマンスと密度の両立)
- 空はHDRIスカイドーム。昼/夕/夜でHDRIを差し替える
- スケール感は現実準拠(天井高2.4m、文机の高さ等)。VRではスケールの狂いが没入を最も壊すので最優先で検証する

## 4. 仕掛け一覧(優先度付き)

### P0 — これがないと完成ではない
1. **移動**:テレポート移動(左スティック or トリガー+レイ)+スナップターン45°
2. **昼夜サイクル**:手元(左手首)のパネルで 朝/昼/夕/夜 を切替。HDRI・ライト・窓からの光が連動
3. **照明のオンオフ**:行灯・デスクランプを個別に点灯/消灯(レイでポイントしてトリガー)
4. **ラジオ**:文机の上のラジオをクリックで再生/停止。音源は `public/audio/*.mp3` を順に再生する仕様にし、**mp3差し替えだけで中身を入れ替えられる**こと(手持ちの音声番組を置ける)
5. **障子/戸の開閉**:書斎⇔縁側の戸がスライドして開く

### P1 — リッチさの核。時間が許す限り実装
6. **天候切替**:晴れ/雨。雨はパーティクル+雨音+庭の池の波紋
7. **掛け軸**:`public/texts/tanka.json` から短歌をランダム表示。クリックで次の一首へ(縦書きレンダリング。JSONは差し替え可能に)
8. **鹿威し**:一定間隔で「コン」と鳴る(3D音源)
9. **焚き火**:夜のみ点火可能。パチパチ音+揺れる光
10. **風鈴**:縁側でかすかに鳴る環境音

### P2 — 余裕があれば
11. 季節切替(庭の楓:緑/紅葉)
12. 本棚の特定の一冊を引くと隠しギミック(内容は実装者の裁量。悪趣味にしない)

## 5. 品質基準・パフォーマンス予算(Quest 2実機基準)

- **72fps維持**を最優先。リッチさとの二択では常にfpsを取る
- ドローコール ≤ 100、三角形数 ≤ 30万、テクスチャは最大2K・KTX2/Basis圧縮
- リアルタイム影は directional light 1灯のみ・低解像度シャドウマップ。それ以外の陰影はテクスチャのAOとライティング設計で稼ぐ
- ポストプロセスは原則使わない(Questでは高コスト)。リッチさはHDRI環境光+PBRマテリアル+色設計で出す
- 初期ロード ≤ 15秒(アセットの遅延ロード可)

## 6. アセット調達方針

- HDRI:Poly Haven(昼・夕・夜の3枚。屋外・日本っぽい光でなくてよい、色温度で調整)
- テクスチャ:ambientCG / Poly Haven(木材・畳代替のファブリック・石・和紙)
- 3Dモデル:基本は**プリミティブ+自前ジオメトリで組む**(文机・本棚・ラジオ等は箱ベースで十分リアルに見せられる)。既製GLBを使う場合もCC0限定
- `scripts/fetch-assets.sh` を作り、URL・ライセンスを `ASSETS.md` に全記録

## 7. 実装体制(サブエージェント分担)

メインセッション(Fable)は全体アーキテクチャ設計・統合・パフォーマンス監査に専念し、実装はサブエージェントに切り出す:

- **Sonnet**:シーンジオメトリ(建物・家具・庭)、アセット取得スクリプト、UIパネル、短歌JSON/縦書き表示
- **Opus**:XRインタラクション基盤(コントローラー入力・テレポート・レイキャスト)、昼夜/天候のライティングシステム、空間オーディオ
- **Fable本体が直接書く**:アーキテクチャの骨格(シーングラフ設計、状態管理、モジュール境界)、最終統合、fps最適化パス

各モジュールはインターフェースを先に定義してから並列実装すること(contract-first)。
